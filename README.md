# LangGraph: Orchestrating Agents and Multi-Agents

Introductory hands-on lesson (1 hour) on **LangGraph**, progressing from a simple agent with RAG to a multi-agent system with distinct roles.

**Domain**: TechMart — Online Electronics Store (customer support center)

## Scenario 1 — FAQ with Simple RAG

### Objective
Introduce the **fundamental concepts** of LangGraph: state, nodes, edges, and conditional edges.

### What the student learns
- Define a `StateGraph` with `TypedDict`
- Create nodes (Python functions)
- Connect with fixed and conditional edges
- Use RAG with FAISS for context retrieval

### Flow

```
[START] → [Classify Question] ─── "policies" → [Search Policies] ─┐
                               └── "products"  → [Search Products]  ──┤
                                                                      └→ [Generate Response] → [END]
```

### Data used
- `data/kb/store_policies.md` — exchange, shipping, payment, warranty, TechPoints policies
- `data/kb/product_faq.md` — notebooks, smartphones, peripherals, smart home, accessories

### Demo questions
1. "What is the deadline to return a product?"
2. "Which notebook do you recommend for programming?"
3. "Do you accept wire transfers? Is there a discount?"
4. "Which monitors do you sell?"
5. "How does the TechPoints program work?"

## Scenario 2 — Multi-Step Agent with Web Search + CSV

### Objective
Show how a graph can have **controlled loops** and compose **multiple tools**.

### What evolves from Scenario 1
- RAG + CSV + Web Search (4 tools)
- Multi-step routing (loop)
- Multiple actions per query
- Data writing (registering tickets)
- `Annotated[list, add]` for state accumulation

### Flow

```
[START] → [Analyze Intent] → [Router] ─┬→ [Search FAQ]        → [Router]
                                        ├→ [Query CSV]         → [Router]
                                        ├→ [Search Web]        → [Router]
                                        ├→ [Register Ticket]   → [Router]
                                        └→ [Generate Response]  → [END]
```

### Data used
- Same FAISS databases from Scenario 1
- `data/csv/tickets.csv` — support tickets with 12 fictional records

### Demo questions
1. "What is the deadline to return a product?" (FAQ only)
2. "What is the status of Maria Silva's ticket?" (CSV only)
3. "My TechBook Essentials has a flickering screen. What is the warranty policy? Is there already a similar ticket?" (FAQ + CSV)
4. "I'm Pedro Henrique (pedro.h@email.com). My TechView 27 QHD monitor arrived with a dead pixel. I want to open a ticket." (CSV write)

## Scenario 3 — Multi-Agent System

### Objective
Demonstrate separation of responsibilities between **specialized agents**.

### Architecture

```
[START] → [PLANNER] → [EXECUTOR] → [REVIEWER] → [END]
```

| Agent | Responsibility |
|---|---|
| **Planner** | Decomposes the request into subtasks |
| **Executor** | Executes each task with the appropriate tool |
| **Reviewer** | Reviews, consolidates, and generates a Markdown report |

### Why does multi-agent make sense here?
When a request requires planning, execution of multiple tools, and completeness review, separating these roles allows:
- Greater clarity of responsibility
- Better prompt quality (each agent is a specialist)
- Ability to iterate and provide feedback

### Complex demo question
> "I'm Maria Silva (maria.silva@email.com). My TechBook Essentials has a flickering screen. I opened a ticket before but the problem persists. I want to know: (1) the exchange policy for this case, (2) the status of my previous ticket, (3) if this model has any manufacturer recall, and (4) I want to open a new urgent ticket."

This question requires:
1. FAQ query → exchange/warranty policy
2. CSV query → Maria Silva's previous ticket
3. Web search → manufacturer recall
4. CSV write → new urgent ticket

## Suggested Exercises

### After Scenario 1
1. Add a 3rd knowledge base and a new route
2. Use embeddings for classification instead of the LLM
3. Add a response validation node

### After Scenario 2
1. Create an automatic prioritization node
2. Query the customer's history before responding
3. Add a loop limit to the state

### After Scenario 3
1. Create a feedback loop: Reviewer → Executor if there are gaps
2. Add a 4th agent for urgency/sentiment classification
3. Export the report as HTML instead of Markdown
